# Trading Agent Refactor Module Contracts

These contracts define how the split modules connect. They are normative for implementation: if a module is changed, its inputs, outputs, persistence, and hard constraints must remain compatible unless the design docs are explicitly revised.

## Global Contracts

### G1. Point-in-Time Data Contract

Every source record and derived decision artifact that can influence trading must carry:

- `event_time`
- `published_at`
- `ingested_at`
- `available_for_decision_at`
- source/provider reference metadata

Consumers may only use records where `available_for_decision_at <= decision_time`. Historical replay, reflection, learning, and strategy evolution must not query latest source tables to reconstruct past decisions.

Legacy insider/Form 4 rows with only `filing_date` precision must use a conservative availability rule: `available_for_decision_at = max(ingested_at, next_market_open_after_filing_date)`. Trading must not assume same-day filing availability without explicit timestamp evidence.

### G2. Side-Effect Ownership Contract

Python orchestration owns persistence, state transitions, paper orders, portfolio mutation, strategy lifecycle mutation, and learning factor activation. LLMs may only return structured proposals through versioned prompts and Pydantic schemas.

### G3. LLM Safety Contract

Any LLM output that can affect trading state must be:

- loaded from a versioned prompt registry
- persisted with prompt/schema version, rendered prompt hash, input context, raw output, parsed output, usage, cost, latency, retry count, and errors
- Pydantic-validated before downstream use
- retried once with validation feedback
- downgraded to the pipeline's safe fallback if validation still fails

### G4. Risk Finality Contract

No scanner, strategy, trading agent, manual request, intraday alert, or strategy evolution output can create exposure without deterministic sizing and risk approval. `RiskManager` is the final gate before paper stock or option order creation.

### G5. Trade Identity Contract

Every trade, watch item, candidate selected for a trading decision, open paper position, option strategy, and hedge overlay must carry exactly one portfolio-pool identity:

- `core_holding`
- `tactical_stock_trade`
- `tactical_option_trade`
- `risk_hedge_overlay`
- `watch_only`

`risk_hedge_overlay` can only be generated by `RiskManager`. `catalyst_watch` and `ordinary_watch` are `watch_type` values under `trade_identity = "watch_only"`.

### G6. Bearish Evidence Contract

Macro-only bearish evidence, valuation-only bearish evidence, RSI-only bearish evidence, or generic "extended stock" reasoning cannot create high-confidence single-name bearish trades. Bearish context can reduce size, block new longs, tighten exits, or downgrade to watch unless direct company-level negative evidence and price/volume confirmation exist.

### G7. Manual Review Contract

Manual ticker requests force evaluation, not trade approval. `review_only` requests can never create paper orders. `paper_trade_eligible` requests can create paper orders only after the same signal, strategy, confidence, sizing, and risk gates as scanner candidates.

### G8. Options Contract

The initial options layer is paper/simulation-only and whitelisted to:

- `long_call`
- `long_put`
- `put_credit_spread`
- `call_credit_spread`
- `long_straddle`
- `long_strangle`

Every option decision must include leg metadata, Greeks, max loss, margin requirement, buying-power effect, event-through-expiry flags, and assignment plan when short option legs can assign. Missing required option data blocks or downgrades the option action.

## Pipeline Contracts

| Producer | Output Contract | Consumers | Hard Constraints |
| --- | --- | --- | --- |
| `UniverseProvider` / `UniverseScanPipeline` | `universe_snapshots`, `universe_symbols`, inclusion/exclusion reasons, active filter version | `SignalPipeline`, UI, replay | Defaults to US common stocks with liquidity/sector filters; manual requests can bypass scanner threshold but not data/liquidity/risk gates. |
| `ManualTickerReviewPipeline` | active `manual_ticker_requests`, request mode, reason, evaluation status | `SignalPipeline`, `StrategyPipeline`, UI, reflection | Forced evaluation only; no trade approval bypass. Requests remain active until dismissed. |
| `SourceIngestionJobs` | normalized source rows plus `source_ingestion_runs` and `provider_request_runs` telemetry | `SignalPipeline`, `HourlySignalRefreshPipeline`, replay | Provider access must use rate limits, cache/freshness gates, budgets, backoff, circuit breakers, and degraded mode. Trading consumes deterministic raw source families including `technical`, `fundamental`, `events_news`, `insider`, `social_macro`, and `option_chain`; it must not consume `research_outputs` or other LLM conclusions on the critical path. |
| `MacroPipeline` | `macro_snapshots` with regime, risk budget multiplier, blocked strategy tags, macro invalidators | `StrategyPipeline`, `TradingPipeline`, `RiskConfigResolver`, UI | Macro does not rank individual stocks and does not create macro-only bearish single-name trades. |
| `PortfolioEventCalendarPipeline` | `calendar_events`, `portfolio_event_risk_assessments` | `SignalPipeline`, `RiskManager`, UI, intraday refresh | UI hides irrelevant low-impact events by default; lookahead window depends on holding horizon/expiry/event importance. |
| Relationship graph | `ticker_relationships`, `peer_baskets`, `theme_taxonomy` | relative strength, read-through, attribution, UI | Relationships must be structured and source-backed, not free-form LLM inference. |
| `SignalPipeline` | pre-open `signal_snapshots` with signal JSON, source refs, freshness, missing/stale fields, PIT audit fields | `StrategyPipeline`, `TradingPipeline`, replay, UI | No hidden prompt-time lookups; missing facts stay missing. Derived signals inherit max input availability time. Supported deterministic snapshot families now include `insider` and `social_macro`; only deep SEC interpretation and transcript/read-through families remain deferred. |
| `StrategyPipeline` | `candidate_scores` per `(ticker, strategy_id)` with explicit `candidate_status`, evidence, horizon, missing signals, invalidators, macro compatibility | `PrimaryStrategySelector`, replay, reflection, UI | Scores are not trade actions. Deferred source families must be marked missing/unsupported rather than backfilled from future data. |
| `PrimaryStrategySelector` | `selected_trades` plus separately persisted `watch_candidates` per ticker/action namespace | `TradeClassifier`, `TradingPipeline`, attribution, UI | Attribution must not drift after selection. Watch outcomes must not be disguised as selected trades or inherit fake expression buckets. |
| `TradeClassifier` | `trade_classifications` for selected trade-path rows only, with trade identity, expression bucket, horizon, exit policy | `TradingPipeline`, `PositionSizer`, `RiskManager`, UI, reflection | `core_holding` requires active approved `portfolio_intent`. New code must not emit `watch_only` trade classifications for non-trade watch rows. |
| `OptionsStrategyLayer` | `option_strategy_decisions` and `option_strategy_legs` for whitelisted paper option structures | `RiskManager`, paper option broker, UI, reflection | Non-whitelisted structures are rejected or downgraded to watch. Option confidence is separate from stock confidence. |
| `TradingPipeline` | Pydantic-validated `trading_decisions` or safe fallbacks with full context snapshot | `PositionSizer`, `RiskManager`, paper brokers, UI, reflection | No broker/order/database side effects beyond persisted decision artifacts. |
| `RiskConfigResolver` | generated `risk_limit_configs` from risk appetite, macro, account state, portfolio composition, trade identity, option risk | `PositionSizer`, `RiskManager`, UI, replay | Hard safety rails remain invariant across conservative/balanced/aggressive presets. |
| `PositionSizer` | `position_sizing_decisions` with base size, volatility adjustment, liquidity cap, factor caps, final size | `RiskManager`, UI, reflection | LLM suggested target weight is advisory only. |
| `RiskManager` | approved/reduced/rejected risk decisions, `portfolio_risk_snapshots`, `portfolio_risk_intents`, `risk_factor_exposures`, hedge overlay decisions | paper brokers, UI, reflection | Final gate for exposure. It consumes `PortfolioHedgePlanner` intent, owns `risk_hedge_overlay` actions, and only materializes hedge overlays after residual post-approval exposure is known. |
| `PaperStockBroker` / option broker | Alpaca-backed stock `paper_orders`/`paper_executions` audit rows, simulated option paper orders, broker rejects, and state transitions | `PortfolioPipeline`, UI, reflection | Paper-only. Stock execution source of truth is Alpaca paper trading; local stock order/execution rows mirror broker state and carry broker/client order ids. Orders are idempotent by trade date/ticker/strategy/action. |
| `PortfolioPipeline` | Alpaca-synced stock positions, option paper positions, and unified margin-account `portfolio_snapshots` | `RiskManager`, intraday refresh, replay, reflection, UI | Stocks and options share one paper margin-account view. PR 6 stock cash/equity/buying power/open positions come from Alpaca paper account sync; option simulation must overlay this account without creating a separate option buying-power pool. |
| `HourlySignalRefreshPipeline` | `intraday_signal_scans`, `intraday_signal_snapshots`, carried-forward baseline fields, deltas | `IntradayRebalancePipeline`, UI, reflection | Does not rerun the full universe scan. Low-frequency fields must be marked carried-forward/stale/missing. |
| `IntradayRebalancePipeline` | risk-gated rebalance proposals and final decisions for material deltas or critical/high alerts | `PositionSizer`, `RiskManager`, paper brokers, UI, reflection | `open_new` disabled by default unless ticker was a morning candidate or approved override. |
| `HistoricalReplayOutcomeEvaluator` | `historical_replay_runs`, `candidate_outcome_evaluations` | reflection, strategy evolution, confidence calibration, UI | Reconstructs candidates from decision-time snapshots. Outcomes use future market data only for measurement, not candidate reconstruction. |
| `ReflectionPipeline` | `daily_reflections`, candidate/observation/risk-tightening learning factors, strategy proposal hints | learning injection, strategy evolution, UI | Reflection failure must not mutate learning factors. Expansionary factors do not become active automatically. |
| `StrategyEvolutionPipeline` | `strategy_proposals`, candidate/shadow strategy definitions, lifecycle transitions | strategy catalog, scanner, UI | New strategies cannot create paper orders until promoted to experimental/active by evidence and risk gates. |
| UI | read-model views plus explicit user config/manual-request mutations | operator | UI must show audit context; it must not bypass pipeline contracts. |

## Change Control

When changing a module contract:

1. Update this file first.
2. Update the relevant design module.
3. Update every implementation module that consumes or produces the changed contract.
4. Update the progress tracker with the contract change and verification command.
